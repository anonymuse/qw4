# Phase 0 Transport Finding Template

Status: template. Fill from real Phase 0 run artifacts only.

Use this template for the first DS5 technical finding:

> Can an M5 Pro coordinator coordinate Qwen3-shaped MoE activation/result movement across two M5 Max workers without interconnect and scheduling overhead dominating decode-shaped work?

This template must support positive, negative, and invalid runs. Do not fill placeholders with estimates unless the section explicitly asks for a planning assumption.

## Publication Guardrails

- This finding is about transport, scheduling, checksums, and simulated MoE traffic.
- This finding does not load Qwen3 weights, run tokenizer or attention, execute expert matmuls, or measure final model quality.
- Loopback, single-host, Wi-Fi fallback, or synthetic-only results are not hardware-cluster claims.
- Positive results justify the next investigation. They are not final model performance claims.
- Negative results are publishable when the artifacts make the limiting factor clear.

- `Proceed`: Transport and scheduler evidence is strong enough to start model metadata inspection and placement simulation.

- `Reduce assumption`: The architecture may proceed only under a lower remote expert-rate assumption.
- `Publish limit`: The measured transport envelope is not compatible with the intended decode-shaped traffic; document the limit and pivot.
- `Rerun`: The run is invalid because the path, artifacts, checksums, or trace coverage are insufficient.

## Required Schema Fields

The reporting path needs the benchmark schema to provide at least:

- run identity: `run_id`, `schema_version`, `git_commit`, `started_at`, `ended_at`;
- scenario identity: `scenario_name`, scenario config path or digest, message sizes, block sizes, transfer counts, remote expert-rate sweep values;
- node context: node IDs, roles, host labels, OS versions, hardware labels, memory, transport addresses, confirmed network path;
- validity context: loopback flag, hardware-path confirmation, clock-sync notes, warmup policy, run validity status;
- latency metrics: node pair, message size bytes, transfer count, p50, p95, p99, jitter or error notes;
- throughput metrics: node pair, block size bytes, transfer count, sustained throughput, checksum cost if measured;
- health and failure events: event timestamp, event type, node, severity, retry count, timeout duration, bytes sent, bytes acknowledged;
- checksum metrics: algorithm, packet count, failure count, failed packet IDs or event IDs when available;
- scheduler metrics: overhead per simulated token, bytes per simulated token, per-layer simulated transport time;
- simulated MoE sensitivity: remote expert rate, local expert rate, predicted upper-bound tokens/sec, and the formula inputs used to compute it;
- decision fields: go/no-go recommendation, invalidation reason if any, and links to generated figures or summary text.
